# alias-mapper CLI — Design Notes (v0)

Status: draft for discussion with Emilio
Author: Max Reese

## Purpose

A command-line tool that takes a bioinformatics input file (GFF, FASTA, etc.)
using one chromosome/scaffold naming convention and rewrites it using another,
based on the alias lookup table generated by the weekly NCBI workflow.

The problem this solves: research files from different sources use different
naming conventions for the same sequences (e.g. `chr1`, `NC_000001.11`,
`CM000663.2`, and `1` can all refer to the same human chromosome). Without
translation, files can't be used together.

## Scope (v1)

- File formats supported: **GFF** and **FASTA**
- Mode: **single file per invocation**
- Naming conventions (from the alias TSV): GenBank, RefSeq, UCSC, Sequence-Name,
  Assigned-Molecule

Out of scope for v1 (notes below):

- Multi-file mode (`--out-dir`)
- BED, VCF, SAM/BAM
- Parallel processing across CPU cores

## Command-line interface (proposed)

```
alias-mapper convert <input> --to <convention> [-o <output>] [options]
```

Examples:

```
# Most common case — auto-detect everything, write to a new file
alias-mapper convert annotations.gff --to ucsc -o annotations.ucsc.gff

# User declares both ends explicitly
alias-mapper convert annotations.gff --from refseq --to ucsc -o out.gff

# User declares which assembly to translate within
alias-mapper convert annotations.gff --assembly GCF_000001405.40 --to ucsc -o out.gff

# Strict mode — error if any sequence name can't be mapped
alias-mapper convert annotations.gff --to ucsc --strict -o out.gff

# Skip unknown sequence names instead of passing them through
alias-mapper convert annotations.gff --to ucsc --skip-unknown -o out.gff
```

### Flags

| Flag             | Required? | Purpose                                                    |
| ---------------- | --------- | ---------------------------------------------------------- |
| `--to`           | yes       | Target naming convention                                   |
| `-o / --output`  | no        | Output path; defaults to `<input>.converted.<ext>`         |
| `--from`         | no        | Source convention; auto-detected if absent                 |
| `--assembly`     | no        | Restrict lookup to one assembly; auto-detected if absent   |
| `--strict`       | no        | Error out on any unmapped sequence name                    |
| `--skip-unknown` | no        | Drop rows whose sequence name isn't in the alias table     |
| `--alias-table`  | no        | Path to an alternative alias TSV (default: latest Release) |

## Design decisions and reasoning

### Naming convention: auto-detect with `--from` override

Most users won't know the source convention with certainty — the file may be
from a collaborator or a public database, or may mix conventions internally.

Auto-detection works by sampling the first ~50 unique sequence names in the
input and checking which convention they best match in the alias table. The
`--from` flag is for power users who already know.

Stretch goal: per-line resolution (handle files that mix conventions on
different rows).

### Assembly scope: auto-detect with `--assembly` override

The same sequence name (e.g. `1`) is ambiguous across species, so the tool
must scope its lookup to one assembly. Auto-detection scores candidate
assemblies by how many of the input's first ~50 unique sequence names exist
in each one. The best match wins. If the top two assemblies are close (i.e.
ambiguous), error out and require `--assembly`.

Cost is negligible — one pass over ~50 names against an in-memory dict.

### Unknown sequence names: warn-but-pass-through by default

If the input has a sequence name the alias table doesn't know (e.g. a custom
contig, a brand-new assembly, or a typo), the default behaviour is to keep
the row unchanged in the output and emit a warning to stderr.

Why not silently skip: silent data loss is a real risk in bioinformatics
pipelines — a missing gene in an output file could propagate into a paper.
Pass-through preserves data and forces the user to consciously notice the
warning.

Why not error out by default: too strict for casual use; most files have a
handful of weird contigs that shouldn't kill the run.

The `--strict` and `--skip-unknown` flags give power users the other two
behaviours when they want them.

### Single-file mode in v1, multi-file deferred

Emilio's "5 GFFs + 1 FASTA" scenario implies real users will want batch
processing. The efficient way to do that is loading the alias TSV once and
processing all files with a single shared lookup dict — which means a future
`--out-dir` multi-file mode is the right next step.

For v1, single-file is the simpler interface and validates the core design.
The tradeoff: a user with 6 files runs the tool 6 times, paying the alias-TSV
load cost each time. Acceptable for v1, worth fixing in v2.

## Architecture sketch

```
[ alias-table.tsv.gz ]
         │ (loaded once at startup)
         ▼
[ in-memory dict: (assembly, source_name) → {refseq, genbank, ucsc, ...} ]
         │
         ▼
[ optional: auto-detect source convention and assembly from sampled input names ]
         │
         ▼
[ stream input file line-by-line ]
         │
         ▼ (for each line)
[ extract sequence name → dict lookup → substitute → write line to output ]
         │
         ▼
[ on EOF: print summary (rows processed, rows unmapped) ]
```

### Why this is efficient

- The alias TSV is loaded **once** at startup into a Python dict — constant-time
  lookups regardless of table size (~1.5M entries).
- Input files are **streamed** line-by-line: memory usage is independent of
  file size. Handles arbitrarily large GFFs and FASTAs.
- Output is written line-by-line: no intermediate buffer.
- Single pass over the input, single dict lookup per row. Linear time.

Naive alternative (re-scanning the TSV for every input row) would be
quadratic and unusable on real-world file sizes.

## Edge cases

Grouped by category. The doc lists what I'd handle in v1 vs. defer.

### Input-file edge cases

| Case                                                  | v1 behaviour                                      |
| ----------------------------------------------------- | ------------------------------------------------- |
| File doesn't exist                                    | Error out cleanly with a clear message            |
| File is empty                                         | Warn, write empty output, exit 0                  |
| File is gzipped (`.gff.gz`)                           | Auto-detect by extension; read transparently      |
| Mixed line endings (`\r\n` vs `\n`)                   | Strip on read; normalise on write                 |
| Malformed line (wrong column count)                   | Warn, pass through unchanged                      |
| Sequence name contains whitespace or weird characters | Look up as-is; if unmapped, warn-and-pass-through |
| File is very large (multi-GB)                         | Streaming handles this; no special logic needed   |

### Naming-convention edge cases

| Case                                                   | v1 behaviour                                             |
| ------------------------------------------------------ | -------------------------------------------------------- |
| Sequence name not in alias table                       | Warn-and-pass-through (or `--strict` / `--skip-unknown`) |
| Sequence name ambiguous across assemblies              | Auto-detection error → require `--assembly`              |
| Accession with version mismatch (`CM000663.1` vs `.2`) | Treat as different sequences; document this clearly      |
| Mitochondrial DNA aliases (`MT`, `chrM`, `chrMT`)      | Should "just work" — these are normal aliases in the TSV |
| Mixed conventions within one file                      | Stretch goal: per-line resolution                        |

### Output edge cases

| Case                                  | v1 behaviour                                                            |
| ------------------------------------- | ----------------------------------------------------------------------- |
| Output file already exists            | Error out unless `--force` is supplied                                  |
| Output path's directory doesn't exist | Create it                                                               |
| Tool fails halfway through writing    | Write to a temp file, rename on success → no partial output left behind |

### Performance edge cases

| Case                                     | v1 behaviour                                                  |
| ---------------------------------------- | ------------------------------------------------------------- |
| Input is multi-GB                        | Streaming handles it                                          |
| Many input files (5 GFFs + 1 FASTA)      | v1: run the tool 6 times. v2: `--out-dir` mode                |
| User has slow disk / network alias-table | Allow `--alias-table` to point at a local pre-downloaded copy |

## Open questions for Emilio

1. Are GFF + FASTA the right v1 scope, or should BED be in v1 too?
2. Where should the alias table live by default — fetched from the latest
   GitHub Release at runtime, or expected to be on disk and pointed at with
   `--alias-table`?
3. How strict should the default behaviour be? Right now I'm proposing
   warn-and-pass-through; an alternative is `--strict` by default with
   `--lenient` to opt out.
4. Should we expose a separate `alias-mapper inspect <file>` subcommand
   that reports the detected source convention and assembly without
   modifying the file? Useful for sanity-checking before a real conversion.
5. Should mapping be lossless-reversible? (i.e. running A→B then B→A should
   give back the original — this constrains how we handle unmapped rows.)
